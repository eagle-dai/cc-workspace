# Humanoid Voice IDF — 给初学者的深入浅出解释

## 这是什么？

这是一份**专利交底书**（Invention Disclosure Form, IDF），不是可运行的代码项目。它描述了一项针对**人形机器人语音交互**的发明：

> **如何让机器人在被用户打断说话后，既能干净利落地停下来，又不会丢失正在执行的任务？**

---

## 它要解决什么问题？

想象你跟一个仓库机器人说话：

### 问题 1：幽灵播报

```
你："去前台接访客"
机器人开始说："好的，我现在出发去前台——"
你（打断）："等等，改去三号会议室"
机器人停了……但 2 秒后突然又冒出："——接访客的路线已规划好" ← 这就是幽灵播报
```

**原因**：机器人的语音是云端生成、通过网络传到本地的。你打断时，已经有一些旧的语音数据包**在路上**了。它们到达后混进播放队列，就会"诈尸"般播出来。

### 问题 2：打断 = 任务全毁

```
你："去前台接访客，然后送文件到会议室"
机器人开始规划路线、调度电梯……
你（打断）："哦对了，避开电梯口区域"
传统方案：整个任务被销毁，从零开始重新规划 ← 巨大浪费
```

**原因**：传统系统把"停止播放语音"和"终止后台任务"绑死了。你只是想**微调**，它却理解成**取消**。

---

## 核心方案：三招解决

### 第一招：给每句话贴"回合标签"

每个语音数据包都带有 `turn_id`（回合编号），就像快递上的订单号：

```
T1 的语音包: { turn_id: "T1", 内容: "好的，出发去前台..." }
T2 的语音包: { turn_id: "T2", 内容: "收到，已改为避开电梯口..." }
```

### 第二招：边缘端状态机 — 四个状态

机器人本地有个"门卫"（状态机），控制哪些语音包能进入播放队列：

```
┌──────┐   开始播放    ┌───────────┐
│ Idle │ ──────────→ │ Active(T1) │    ← 正常播放 T1
└──────┘              └───────────┘
                           │ 用户打断！
                           ▼
                    ┌──────────────┐
                    │ Cancelled(T1)│    ← 立刻停止+清空播放队列（过渡态）
                    └──────────────┘
                           │ 自动进入
                           ▼
                  ┌──────────────────┐
                  │AwaitingResume(T1)│  ← 锁定！拒绝一切 T1 的包
                  └──────────────────┘
                           │ 收到云端合法的恢复消息
                           ▼
                    ┌───────────┐
                    │ Active(T2) │     ← 新回合，正常播放
                    └───────────┘
```

**关键**：在 `AwaitingResume` 状态下，迟到的 T1 语音包到达时，门卫看一眼 `turn_id`——"T1？不让进！"——直接丢弃。**幽灵播报问题解决。**

### 第三招：增量约束更新 — 不销毁，只修补

打断后，机器人**不会**告诉云端"取消任务"，而是说：

```
"用户新增了一个约束：避开电梯口区域。其他不变，请局部修改。"
```

云端收到后：
- 保留已有的任务上下文（去前台、接访客、送文件）
- 只对受影响的部分重新规划（路线避开电梯口）
- 生成新回合 T2 的语音，附带恢复控制消息

```json
{
  "turn_id": "T2",
  "parent_turn_id": "T1",
  "message_type": "resume_ctrl"
}
```

机器人门卫验证 `parent_turn_id == T1`（当前锁定的回合），通过！解锁，开始播放 T2。**任务连续性问题解决。**

---

## 完整流程一图流

```
时间线 →

用户:   "去前台接访客，送文件到会议室"          "避开电梯口"
         │                                      │
机器人:  [Idle]→[Active T1] 播放中...    打断→[Cancelled]→[AwaitingResume T1]
         │                                      │
云端:    规划任务，下发 T1 音频 ───→  ···残余T1包···→  到达，被丢弃 ✗
                                                │
         增量约束上传: {avoid: "电梯口"}  ←────────┘
         │
云端:    保留上下文，局部重规划
         │
         下发 resume_ctrl {T2, parent=T1} ──→ 门卫验证通过 → [Active T2]
         下发 T2 音频 ─────────────────────→ 正常播放 ✓
```

---

## 为什么这个方案巧妙？

| 维度 | 传统方案 | 本发明 |
|------|---------|--------|
| 幽灵播报 | 靠"停止播放"，迟到的包仍可能混入 | **入队前按 turn_id 物理丢弃**，从源头隔绝 |
| 打断语义 | 打断 = 终止任务 | 打断 = 修正约束，任务继续 |
| 云端算力 | 每次打断都从零开始 | 局部重规划，复用已有计算图 |
| 安全性 | 恢复播放无验证 | `parent_turn_id` 回合链校验 |
| 容错 | 无 | 超时 fail-safe，防止系统死锁 |

---

## 一句话总结

> 给语音包贴上"回合编号"，打断时锁住旧编号的入口，同时告诉云端"别停，只是改改"——这就是整个发明的精髓。
